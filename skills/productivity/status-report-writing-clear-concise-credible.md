Write weekly status reports that stakeholders actually read using a consistent 4-section structure.

## When to use
Weekly project status reports, monthly team updates, exec briefings.

## Template
```markdown
## [Project] Status — Week of [Date]
**Overall status**: GREEN / YELLOW / RED
(GREEN = on track; YELLOW = risk, monitoring; RED = off track, needs help)

### This week: what moved
- [Outcome 1 — specific and measurable]
- [Outcome 2]

### Next week: what's planned
- [Outcome 1]
- [Outcome 2]

### Risks / issues
| Risk/Issue | Impact | Mitigation | Owner |
|------------|--------|------------|-------|
| | | | |

### Metrics (if applicable)
| Metric | Last week | This week | Target | Trend |
|--------|-----------|-----------|--------|-------|
| | | | | |

### Decisions needed from stakeholders
- [Decision] — needed by [date]
```

## Tips
- RAG status (Red/Amber/Green) must be honest. Yellow that's actually Red destroys trust when things blow up.
- "This week" should be outcomes, not activities. "Reviewed design" is activity; "Design approved, ready for build" is outcome.
- Keep it under one page. Longer = less read. Use attachments for detail.
- Send at the same time every week — predictability builds trust that the project is under control.
